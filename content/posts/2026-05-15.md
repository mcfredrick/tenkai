## Research Worth Reading

- **[GraphBit](https://arxiv.org/abs/2605.13848)** — Replaces prompted orchestration in agent frameworks with explicit DAG-based workflow definitions. No more hallucinated routing or infinite loops from the model "winging it." Deterministic execution is underrated. 📄
- **[SPIN](https://arxiv.org/abs/2605.14051)** — Planning wrapper that combines validated DAG planning with prefix-based execution to stop LLM planners from generating structurally broken or wildly bloated workflows. Targets the brittle-failure problem in production agent systems.
- **[EvolveMem](https://arxiv.org/abs/2605.13941)** — Co-evolves stored content *and* the retrieval/scoring infrastructure in agent memory. The core insight: your scoring functions shouldn't be frozen at deploy while your content keeps changing. AutoResearch-driven adaptation. Smart.
- **[TabPFN-3](https://arxiv.org/abs/2605.13986)** — Scales tabular foundation models to 1M-row datasets with substantially lower training and inference cost. If you've been ignoring foundation models for tabular data because of scale limits, this one's for you. 🔥
- **[HodgeCover](https://arxiv.org/abs/2605.13997)** — Tackles an irreducible cycle obstruction in MoE compression where pairwise-compatible experts form incompatible triples. Higher-order topological coverage framework. Niche but real if you care about efficient MoE inference.
- **[PREPING](https://arxiv.org/abs/2605.13880)** — Pre-task memory construction for agents. Addresses the cold-start problem: what do you stuff in agent memory *before* it has any task experience? No curated demos, no post-deploy bootstrapping required.

## Today's Synthesis

If you're building agent systems headed for production, today's reads converge on one theme: stop letting LLMs wing the control flow and start treating your data layer as infrastructure, not an afterthought. [GraphBit](https://arxiv.org/abs/2605.13848) and [SPIN](https://arxiv.org/abs/2605.14051) both argue for replacing prompted orchestration with explicit DAG-based planning — one at the workflow definition level, the other at execution validation. Together they sketch a clear pattern: define routing as structure, not suggestion. But deterministic execution only helps if the data you're retrieving is actually any good. That's where [corpulse](https://pypi.org/project/corpulse/0.1.0/) fits — it gives you instrumentation for the corpus feeding your RAG pipeline, tracking quality, coverage, and drift over time. The concrete play: wire your agent orchestration through DAG-defined workflows, then instrument the retrieval corpus with corpulse so you can distinguish routing bugs from data rot when things break. Two different failure modes, two different fixes, and both are now actionable.
